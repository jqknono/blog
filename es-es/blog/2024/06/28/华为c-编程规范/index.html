C++语言编程规范

目的

规则并不是完美的，通过禁止在特定情况下有用的特性，可能会对代码实现造成影响。但是我们制定规则的目的“为了大多数程序员可以得到更多的好处”， 如果在团队运作中认为某个规则无法遵循，希望可以共同改进该规则。 参考该规范之前，希望您具有相应的C++语言基础能力，而不是通过该文档来学习C++语言。

  1. 了解C++语言的ISO标准；
  2. 熟知C++语言的基本语言特性，包括C++ 03/11/14/17相关特性；
  3. 了解C++语言的标准库；

总体原则

代码需要在保证功能正确的前提下，满足可读、可维护、安全、可靠、可测试、高效、可移植的特征要求。

重点关注

  1. 约定C++语言的编程风格，比如命名，排版等。
  2. C++语言的模块化设计，如何设计头文件，类，接口和函数。
  3. C++语言相关特性的优秀实践，比如常量，类型转换，资源管理，模板等。
  4. 现代C++语言的优秀实践，包括C++11/14/17中可以提高代码可维护性，提高代码可靠性的相关约定。
  5. 本规范优先适于用C++17版本。

约定

规则：编程时必须遵守的约定(must)

建议：编程时应该遵守的约定(should)

本规范适用通用C++标准, 如果没有特定的标准版本，适用所有的版本(C++03/11/14/17)。

例外

无论是’规则’还是’建议’，都必须理解该条目这么规定的原因，并努力遵守。 但是，有些规则和建议可能会有例外。

在不违背总体原则，经过充分考虑，有充足的理由的前提下，可以适当违背规范中约定。 例外破坏了代码的一致性，请尽量避免。‘规则’的例外应该是极少的。

下列情况，应风格一致性原则优先： 修改外部开源代码、第三方代码时，应该遵守开源代码、第三方代码已有规范，保持风格统一。

2 命名

通用命名

驼峰风格(CamelCase) 大小写字母混用，单词连在一起，不同单词间通过单词首字母大写来分开。 按连接后的首字母是否大写，又分: 大驼峰(UpperCamelCase)和小驼峰(lowerCamelCase)

类型命名风格
类类型，结构体类型，枚举类型，联合体类型等类型定义， 作用域名称大驼峰
函数(包括全局函数，作用域函数，成员函数)大驼峰
全局变量(包括全局和命名空间域下的变量，类静态变量)，局部变量，函数参数，类、结构体和联合体中的成员变量小驼峰
宏，常量(const)，枚举值，goto 标签全大写，下划线分割

注意： 上表中__常量__是指全局作用域、namespace域、类的静态成员域下，以 const或constexpr 修饰的基本数据类型、枚举、字符串类型的变量，不包括数组和其他类型变量。 上表中__变量__是指除常量定义以外的其他变量，均使用小驼峰风格。

文件命名

规则2.2.1 C++文件以.cpp结尾，头文件以.h结尾

我们推荐使用.h作为头文件的后缀，这样头文件可以直接兼容C和C++。 我们推荐使用.cpp作为实现文件的后缀，这样可以直接区分C++代码，而不是C代码。

目前业界还有一些其他的后缀的表示方法：

  • 头文件： .hh, .hpp, .hxx
  • cpp文件：.cc, .cxx, .c

如果当前项目组使用了某种特定的后缀，那么可以继续使用，但是请保持风格统一。 但是对于本文档，我们默认使用.h和.cpp作为后缀。

规则2.2.2 C++文件名和类名保持一致

C++的头文件和cpp文件名和类名保持一致，使用下划线小写风格。

如果有一个类叫DatabaseConnection，那么对应的文件名：

  • database_connection.h
  • database_connection.cpp

结构体，命名空间，枚举等定义的文件名类似。

函数命名

函数命名统一使用大驼峰风格，一般采用动词或者动宾结构。

class List {
public:
	void AddElement(const Element& element);
	Element GetElement(const unsigned int index) const;
	bool IsEmpty() const;
};

namespace Utils {
    void DeleteUser();
}

类型命名采用大驼峰命名风格。 所有类型命名——类、结构体、联合体、类型定义（typedef）、枚举——使用相同约定，例如：

// classes, structs and unions
class UrlTable { ...
class UrlTableTester { ...
struct UrlTableProperties { ...

// typedefs
typedef std::map<std::string, UrlTableProperties*> PropertiesMap;

// enums
enum UrlTableErrors { ...

对于命名空间的命名，建议使用大驼峰：

// namespace
namespace OsUtils {

}

建议2.4.1 避免滥用 typedef或者#define 对基本类型起别名

除有明确的必要性，否则不要用 typedef/#define 对基本数据类型进行重定义。 优先使用<cstdint>头文件中的基本类型：

通用变量命名采用小驼峰，包括全局变量，函数形参，局部变量，成员变量。

std::string tableName;  // Good: 推荐此风格
std::string tablename;  // Bad: 禁止此风格
std::string path;       // Good: 只有一个单词时，小驼峰为全小写

规则2.5.1 全局变量应增加 ‘g_’ 前缀，静态变量命名不需要加特殊前缀

全局变量是应当尽量少使用的，使用时应特别注意，所以加上前缀用于视觉上的突出，促使开发人员对这些变量的使用更加小心。

  • 全局静态变量命名与全局变量相同。
  • 函数内的静态变量命名与普通局部变量相同。
  • 类的静态成员变量和普通成员变量相同。
int g_activeConnectCount;

void Func()
{
    static int packetCount = 0; 
    ...
}

规则2.5.2 类的成员变量命名以小驼峰加后下划线组成

class Foo {
private:
    std::string fileName_;   // 添加_后缀，类似于K&R命名风格
};

对于struct/union的成员变量，仍采用小驼峰不加后缀的命名方式，与局部变量命名风格一致。

#define MAX(a, b)   (((a) < (b)) ? (b) : (a)) // 仅对宏命名举例，并不推荐用宏实现此类功能

enum TintColor {    // 注意，枚举类型名用大驼峰，其下面的取值是全大写，下划线相连
    RED,

3 格式

行宽

规则3.1.1 行宽不超过 120 个字符

建议每行字符数不要超过 120 个。如果超过120个字符，请选择合理的方式进行换行。

例外:

  • 如果一行注释包含了超过120 个字符的命令或URL，则可以保持一行，以方便复制、粘贴和通过grep查找；
  • 包含长路径的 #include 语句可以超出120 个字符，但是也需要尽量避免；
  • 编译预处理中的error信息可以超出一行。 预处理的 error 信息在一行便于阅读和理解，即使超过 120 个字符。
#ifndef XXX_YYY_ZZZ
#error Header aaaa/bbbb/cccc/abc.h must only be included after xxxx/yyyy/zzzz/xyz.h, because xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
#endif

缩进

规则3.2.1 使用空格进行缩进，每次缩进4个空格

只允许使用空格(space)进行缩进，每次缩进为 4 个空格。不允许使用Tab符进行缩进。 当前几乎所有的集成开发环境（IDE）都支持配置将Tab符自动扩展为4空格输入；请配置你的IDE支持使用空格进行缩进。

大括号

规则3.3.1 使用 K&R 缩进风格

K&R风格 换行时，函数（不包括lambda表达式）左大括号另起一行放行首，并独占一行；其他左大括号跟随语句放行末。 右大括号独占一行，除非后面跟着同一语句的剩余部分，如 do 语句中的 while，或者 if 语句的 else/else if，或者逗号、分号。

如：

struct MyType {     // 跟随语句放行末，前置1空格
    ...
};

int Foo(int a)
{                   // 函数左大括号独占一行，放行首
    if (...) {
        ...
    } else {
        ...
    }
}

推荐这种风格的理由：

  • 代码更紧凑；
  • 相比另起一行，放行末使代码阅读节奏感上更连续；
  • 符合后来语言的习惯，符合业界主流习惯；
  • 现代集成开发环境（IDE）都具有代码缩进对齐显示的辅助功能，大括号放在行尾并不会对缩进和范围产生理解上的影响。

对于空函数体，可以将大括号放在同一行：

class MyClass {
public:
    MyClass() : value_(0) {}
   
private:
    int value_;
};

函数声明和定义

规则3.4.1 函数声明和定义的返回类型和函数名在同一行；函数参数列表超出行宽时要换行并合理对齐

在声明和定义函数的时候，函数的返回值类型应该和函数名在同一行；如果行宽度允许，函数参数也应该放在一行；否则，函数参数应该换行，并进行合理对齐。 参数列表的左圆括号总是和函数名在同一行，不要单独一行；右圆括号总是跟随最后一个参数。

换行举例：

ReturnType FunctionName(ArgType paramName1, ArgType paramName2)   // Good：全在同一行
{
    ...
}

ReturnType VeryVeryVeryLongFunctionName(ArgType paramName1,     // 行宽不满足所有参数，进行换行
                                        ArgType paramName2,     // Good：和上一行参数对齐
                                        ArgType paramName3)
{
    ...
}

ReturnType LongFunctionName(ArgType paramName1, ArgType paramName2, // 行宽限制，进行换行
    ArgType paramName3, ArgType paramName4, ArgType paramName5)     // Good: 换行后 4 空格缩进
{
    ...
}

ReturnType ReallyReallyReallyReallyLongFunctionName(            // 行宽不满足第1个参数，直接换行
    ArgType paramName1, ArgType paramName2, ArgType paramName3) // Good: 换行后 4 空格缩进
{
    ...
}

函数调用

规则3.5.1 函数调用入参列表应放在一行，超出行宽换行时，保持参数进行合理对齐

函数调用时，函数参数列表放在一行。参数列表如果超过行宽，需要换行并进行合理的参数对齐。 左圆括号总是跟函数名，右圆括号总是跟最后一个参数。

换行举例：

ReturnType result = FunctionName(paramName1, paramName2);   // Good：函数参数放在一行

ReturnType result = FunctionName(paramName1,
                                 paramName2,                // Good：保持与上方参数对齐
                                 paramName3);

ReturnType result = FunctionName(paramName1, paramName2,
    paramName3, paramName4, paramName5);                    // Good：参数换行，4 空格缩进

ReturnType result = VeryVeryVeryLongFunctionName(           // 行宽不满足第1个参数，直接换行
    paramName1, paramName2, paramName3);                    // 换行后，4 空格缩进

如果函数调用的参数存在内在关联性，按照可理解性优先于格式排版要求，对参数进行合理分组换行。

// Good：每行的参数代表一组相关性较强的数据结构，放在一行便于理解
int result = DealWithStructureLikeParams(left.x, left.y,     // 表示一组相关参数
                                         right.x, right.y);  // 表示另外一组相关参数

if语句

规则3.6.1 if语句必须要使用大括号

我们要求if语句都需要使用大括号，即便只有一条语句。

理由：

  • 代码逻辑直观，易读；
  • 在已有条件语句代码上增加新代码时不容易出错；
  • 对于在if语句中使用函数式宏时，有大括号保护不易出错（如果宏定义时遗漏了大括号）。
if (objectIsNotExist) {         // Good：单行条件语句也加大括号
    return CreateNewObject();
}

规则3.6.2 禁止 if/else/else if 写在同一行

条件语句中，若有多个分支，应该写在不同行。

如下是正确的写法：

if (someConditions) {
    DoSomething();
    ...
} else {  // Good: else 与 if 在不同行
    ...
}

下面是不符合规范的案例：

if (someConditions) { ... } else { ... } // Bad: else 与 if 在同一行

循环语句

规则3.7.1 循环语句必须使用大括号

和条件表达式类似，我们要求for/while循环语句必须加上大括号，即便循环体是空的，或循环语句只有一条。

for (int i = 0; i < someRange; i++) {   // Good: 使用了大括号
    DoSomething();
}

while (condition) { }   // Good：循环体是空，使用大括号

while (condition) {
    continue;           // Good：continue 表示空逻辑，使用大括号
}

坏的例子：

for (int i = 0; i < someRange; i++)
    DoSomething();      // Bad: 应该加上括号

while (condition);      // Bad：使用分号容易让人误解是while语句中的一部分

switch语句

规则3.8.1 switch 语句的 case/default 要缩进一层

switch 语句的缩进风格如下：

switch (var) {
    case 0:             // Good: 缩进
        DoSomething1(); // Good: 缩进
        break;
    case 1: {           // Good: 带大括号格式
        DoSomething2();
        break;
    }
    default:
        break;
}

switch (var) {
case 0:                 // Bad: case 未缩进
    DoSomething();
    break;
default:                // Bad: default 未缩进
    break;
}

表达式

建议3.9.1 表达式换行要保持换行的一致性，运算符放行末

较长的表达式，不满足行宽要求的时候，需要在适当的地方换行。一般在较低优先级运算符或连接符后面截断，运算符或连接符放在行末。 运算符、连接符放在行末，表示“未结束，后续还有”。 例：

// 假设下面第一行已经不满足行宽要求

if ((currentValue > threshold) &&  // Good：换行后，逻辑操作符放在行尾
    someCondition) {
    DoSomething();
    ...
}

int result = reallyReallyLongVariableName1 +    // Good
             reallyReallyLongVariableName2;

表达式换行后，注意保持合理对齐，或者4空格缩进。参考下面例子

int sum = longVariableName1 + longVariableName2 + longVariableName3 +
    longVariableName4 + longVariableName5 + longVariableName6;         // Good: 4空格缩进

int sum = longVariableName1 + longVariableName2 + longVariableName3 +
          longVariableName4 + longVariableName5 + longVariableName6;   // Good: 保持对齐

变量赋值

规则3.10.1 多个变量定义和赋值语句不允许写在一行

每行只有一个变量初始化的语句，更容易阅读和理解。

int maxCount = 10;
bool isCompleted = false;

下面是不符合规范的示例：

int maxCount = 10; bool isCompleted = false; // Bad：多个变量初始化需要分开放在多行，每行一个变量初始化
int x, y = 0;  // Bad：多个变量定义需要分行，每行一个

int pointX;
int pointY;
...
pointX = 1; pointY = 2;  // Bad：多个变量赋值语句放同一行

例外：for 循环头、if 初始化语句（C++17）、结构化绑定语句（C++17）中可以声明和初始化多个变量。这些语句中的多个变量声明有较强关联，如果强行分成多行会带来作用域不一致，声明和初始化割裂等问题。

初始化

初始化包括结构体、联合体、及数组的初始化

规则3.11.1 初始化换行时要有缩进，并进行合理对齐

结构体或数组初始化时，如果换行应保持4空格缩进。 从可读性角度出发，选择换行点和对齐位置。

const int rank[] = {
    16, 16, 16, 16, 32, 32, 32, 32,
    64, 64, 64, 64, 32, 32, 32, 32
};

指针与引用

建议3.12.1 指针类型"*“跟随变量名或者类型，不要两边都留有或者都没有空格

指针命名: *靠左靠右都可以，但是不要两边都有或者都没有空格。

int* p = nullptr;  // Good
int *p = nullptr;  // Good

int*p = nullptr;   // Bad
int * p = nullptr; // Bad

例外：当变量被 const 修饰时，"*” 无法跟随变量，此时也不要跟随类型。

const char * const VERSION = "V100";

建议3.12.2 引用类型"&“跟随变量名或者类型，不要两边都留有或者都没有空格

引用命名：&靠左靠右都可以，但是不要两边都有或者都没有空格。

int i = 8;

int& p = i;     // Good
int &p = i;     // Good
int*& rp = pi;  // Good，指针的引用，*& 一起跟随类型
int *&rp = pi;  // Good，指针的引用，*& 一起跟随变量名
int* &rp = pi;  // Good，指针的引用，* 跟随类型，& 跟随变量名

int & p = i;    // Bad
int&p = i;      // Bad

编译预处理

规则3.13.1 编译预处理的”#“统一放在行首，嵌套编译预处理语句时，”#“可以进行缩进

编译预处理的”#“统一放在行首，即使编译预处理的代码是嵌入在函数体中的，”#“也应该放在行首。

规则3.13.2 避免使用宏

宏会忽略作用域，类型系统以及各种规则，容易引发问题。应尽量避免使用宏定义，如果必须使用宏，要保证证宏名的唯一性。 在C++中，有许多方式来避免使用宏：

  • 用const或enum定义易于理解的常量
  • 用namespace避免名字冲突
  • 用inline函数避免函数调用的开销
  • 用template函数来处理多种类型

在文件头保护宏、条件编译、日志记录等必要场景中可以使用宏。

规则3.13.3 禁止使用宏来表示常量

宏是简单的文本替换，在预处理阶段完成，运行报错时直接报相应的值；跟踪调试时也是显示值，而不是宏名； 宏没有类型检查，不安全； 宏没有作用域。

规则3.13.4 禁止使用函数式宏

宏义函数式宏前，应考虑能否用函数替代。对于可替代场景，建议用函数替代宏。 函数式宏的缺点如下：

  • 函数式宏缺乏类型检查，不如函数调用检查严格
  • 宏展开时宏参数不求值，可能会产生非预期结果
  • 宏没有独立的作用域
  • 宏的技巧性太强，例如#的用法和无处不在的括号，影响可读性
  • 在特定场景中必须用编译器对宏的扩展语法，如GCC的statement expression，影响可移植性
  • 宏在预编译阶段展开后，在期后编译、链接和调试时都不可见；而且包含多行的宏会展开为一行。函数式宏难以调试、难以打断点，不利于定位问题
  • 对于包含大量语句的宏，在每个调用点都要展开。如果调用点很多，会造成代码空间的膨胀

函数没有宏的上述缺点。但是，函数相比宏，最大的劣势是执行效率不高（增加函数调用的开销和编译器优化的难度）。 为此，可以在必要时使用内联函数。内联函数跟宏类似，也是在调用点展开。不同之处在于内联函数是在编译时展开。

内联函数兼具函数和宏的优点：

  • 内联函数执行严格的类型检查
  • 内联函数的参数求值只会进行一次
  • 内联函数就地展开，没有函数调用的开销
  • 内联函数比函数优化得更好

对于性能要求高的产品代码，可以考虑用内联函数代替函数。

例外： 在日志记录场景中，需要通过函数式宏保持调用点的文件名（FILE）、行号（LINE）等信息。

空格和空行

规则3.14.1 水平空格应该突出关键字和重要信息，避免不必要的留白

水平空格应该突出关键字和重要信息，每行代码尾部不要加空格。总体规则如下：

  • if, switch, case, do, while, for等关键字之后加空格；
  • 小括号内部的两侧，不要加空格；
  • 大括号内部两侧有无空格，左右必须保持一致；
  • 一元操作符（& * + ‐ ~ !）之后不要加空格；
  • 二元操作符（= + ‐ < > * / % | & ^ <= >= == != ）左右两侧加空格
  • 三目运算符（? :）符号两侧均需要空格
  • 前置和后置的自增、自减（++ –）和变量之间不加空格
  • 结构体成员操作符（. ->）前后不加空格
  • 逗号(,)前面不加空格，后面增加空格
  • 对于模板和类型转换(<>)和类型之间不要添加空格
  • 域操作符(::)前后不要添加空格
  • 冒号(:)前后根据情况来判断是否要添加空格

常规情况：

void Foo(int b) {  // Good：大括号前应该留空格

int i = 0;  // Good：变量初始化时，=前后应该有空格，分号前面不要留空格

int buf[BUF_SIZE] = {0};    // Good：大括号内两侧都无空格

函数定义和函数调用：

int result = Foo(arg1,arg2);
                    ^    // Bad: 逗号后面需要增加空格

int result = Foo( arg1, arg2 );
                 ^          ^  // Bad: 函数参数列表的左括号后面不应该有空格，右括号前面不应该有空格

指针和取地址

x = *p;     // Good：*操作符和指针p之间不加空格
p = &x;     // Good：&操作符和变量x之间不加空格
x = r.y;    // Good：通过.访问成员变量时不加空格
x = r->y;   // Good：通过->访问成员变量时不加空格

操作符：

x = 0;   // Good：赋值操作的=前后都要加空格
x = -5;  // Good：负数的符号和数值之前不要加空格
++x;     // Good：前置和后置的++/--和变量之间不要加空格
x--;

if (x && !y)  // Good：布尔操作符前后要加上空格，！操作和变量之间不要空格
v = w * x + y / z;  // Good：二元操作符前后要加空格
v = w * (x + z);    // Good：括号内的表达式前后不需要加空格

int a = (x < y) ? x : y;  // Good: 三目运算符， ？和：前后需要添加空格

循环和条件语句：

if (condition) {  // Good：if关键字和括号之间加空格，括号内条件语句前后不加空格
    ...
} else {           // Good：else关键字和大括号之间加空格
    ...
}

while (condition) {}   // Good：while关键字和括号之间加空格，括号内条件语句前后不加空格

for (int i = 0; i < someRange; ++i) {  // Good：for关键字和括号之间加空格，分号之后加空格
    ...
}

switch (condition) {  // Good: switch 关键字后面有1空格
    case 0:     // Good：case语句条件和冒号之间不加空格
        ...
        break;
    ...
    default:
        ...
        break;
}

模板和转换

// 尖括号(< and >) 不与空格紧邻, < 前没有空格, > 和 ( 之间也没有.
vector<string> x;
y = static_cast<char*>(x);

// 在类型与指针操作符之间留空格也可以, 但要保持一致.
vector<char *> x;

域操作符

std::cout;    // Good: 命名空间访问，不要留空格

int MyClass::GetValue() const {}  // Good: 对于成员函数定义，不要留空格

冒号

// 添加空格的场景

// Good: 类的派生需要留有空格
class Sub : public Base {
   
};

// 构造函数初始化列表需要留有空格
MyClass::MyClass(int var) : someVar_(var)
{
    DoSomething();
}

// 位域表示也留有空格
struct XX {
    char a : 4;    
    char b : 5;    
    char c : 4;
};

// 不添加空格的场景

// Good: 对于public:, private:这种类访问权限的冒号不用添加空格
class MyClass {
public:
    MyClass(int var);
private:
    int someVar_;
};

// 对于switch-case的case和default后面的冒号不用添加空格
switch (value)
{
    case 1:
        DoSomething();
        break;
    default:
        break;
}

注意：当前的集成开发环境（IDE）可以设置删除行尾的空格，请正确配置。

建议3.14.1 合理安排空行，保持代码紧凑

减少不必要的空行，可以显示更多的代码，方便代码阅读。下面有一些建议遵守的规则：

  • 根据上下内容的相关程度，合理安排空行；
  • 函数内部、类型定义内部、宏内部、初始化表达式内部，不使用连续空行
  • 不使用连续 3 个空行，或更多
  • 大括号内的代码块行首之前和行尾之后不要加空行，但namespace的大括号内不作要求。
int Foo()
{
    ...
}



int Bar()  // Bad：最多使用连续2个空行。
{
    ...
}


if (...) {
        // Bad：大括号内的代码块行首不要加入空行
    ...
        // Bad：大括号内的代码块行尾不要加入空行
}

int Foo(...)
{
        // Bad：函数体内行首不要加空行
    ...
}

规则3.15.1 类访问控制块的声明依次序是 public:, protected:, private:，缩进和 class 关键字对齐

class MyClass : public BaseClass {
public:      // 注意没有缩进
    MyClass();  // 标准的4空格缩进
    explicit MyClass(int var);
    ~MyClass() {}

    void SomeFunction();
    void SomeFunctionThatDoesNothing()
    {
    }

    void SetVar(int var) { someVar_ = var; }
    int GetVar() const { return someVar_; }

private:
    bool SomeInternalFunction();

    int someVar_;
    int someOtherVar_;
};

在各个部分中，建议将类似的声明放在一起, 并且建议以如下的顺序: 类型 (包括 typedef, using 和嵌套的结构体与类), 常量, 工厂函数, 构造函数, 赋值运算符, 析构函数, 其它成员函数, 数据成员。

规则3.15.2 构造函数初始化列表放在同一行或按四格缩进并排多行

// 如果所有变量能放在同一行:
MyClass::MyClass(int var) : someVar_(var)
{
    DoSomething();
}

// 如果不能放在同一行,
// 必须置于冒号后, 并缩进4个空格
MyClass::MyClass(int var)
    : someVar_(var), someOtherVar_(var + 1)  // Good: 逗号后面留有空格
{
    DoSomething();
}

// 如果初始化列表需要置于多行, 需要逐行对齐
MyClass::MyClass(int var)
    : someVar_(var),             // 缩进4个空格
      someOtherVar_(var + 1)
{

}

4 注释

一般的，尽量通过清晰的架构逻辑，好的符号命名来提高代码可读性；需要的时候，才辅以注释说明。 注释是为了帮助阅读者快速读懂代码，所以要从读者的角度出发，按需注释。

注释内容要简洁、明了、无二义性，信息全面且不冗余。

注释跟代码一样重要。 写注释时要换位思考，用注释去表达此时读者真正需要的信息。在代码的功能、意图层次上进行注释，即注释解释代码难以表达的意图，不要重复代码信息。 修改代码时，也要保证其相关注释的一致性。只改代码，不改注释是一种不文明行为，破坏了代码与注释的一致性，让阅读者迷惑、费解，甚至误解。

使用英文进行注释。

注释风格

在 C++ 代码中，使用 /* */和 // 都是可以的。 按注释的目的和位置，注释可分为不同的类型，如文件头注释、函数头注释、代码注释等等； 同一类型的注释应该保持统一的风格。

注意：本文示例代码中，大量使用 ‘//’ 后置注释只是为了更精确的描述问题，并不代表这种注释风格更好。

函数头注释

规则4.3.1 公有（public）函数必须编写函数头注释

公有函数属于类对外提供的接口，调用者需要了解函数的功能、参数的取值范围、返回的结果、注意事项等信息才能正常使用。 特别是参数的取值范围、返回的结果、注意事项等都无法做到自注示，需要编写函数头注释辅助说明。

规则4.3.2 禁止空有格式的函数头注释

并不是所有的函数都需要函数头注释； 函数签名无法表达的信息，加函数头注释辅助说明;

函数头注释统一放在函数声明或定义上方，使用如下风格之一： 使用//写函数头

// 单行函数头
int Func1(void);

// 多行函数头
// 第二行
int Func2(void);

使用/* */写函数头

/* 单行函数头 */
int Func1(void);

/*
 * 另一种单行函数头
 */
int Func2(void);

/*
 * 多行函数头
 * 第二行
 */
int Func3(void);

函数尽量通过函数名自注释，按需写函数头注释。 不要写无用、信息冗余的函数头；不要写空有格式的函数头。

函数头注释内容可选，但不限于：功能说明、返回值，性能约束、用法、内存约定、算法实现、可重入的要求等等。 模块对外头文件中的函数接口声明，其函数头注释，应当将重要、有用的信息表达清楚。

例：

/*
 * 返回实际写入的字节数，-1表示写入失败
 * 注意，内存 buf 由调用者负责释放
 */
int WriteString(const char *buf, int len);

坏的例子：

/*
 * 函数名：WriteString
 * 功能：写入字符串
 * 参数：
 * 返回值：
 */
int WriteString(const char *buf, int len);

上面例子中的问题：

  • 参数、返回值，空有格式没内容
  • 函数名信息冗余
  • 关键的 buf 由谁释放没有说清楚

代码注释

规则4.4.1 代码注释放于对应代码的上方或右边

规则4.4.2 注释符与注释内容间要有1空格；右置注释与前面代码至少1空格

代码上方的注释，应该保持对应代码一样的缩进。 选择并统一使用如下风格之一： 使用//


// 这是单行注释
DoSomething();

// 这是多行注释
// 第二行
DoSomething();

使用/*' '*/

/* 这是单行注释 */
DoSomething();

/*
 * 另一种方式的多行注释
 * 第二行
 */
DoSomething();

代码右边的注释，与代码之间，至少留1空格，建议不超过4空格。 通常使用扩展后的 TAB 键即可实现 1-4 空格的缩进。

选择并统一使用如下风格之一：

int foo = 100;  // 放右边的注释
int bar = 200;  /* 放右边的注释 */

右置格式在适当的时候，上下对齐会更美观。 对齐后的注释，离左边代码最近的那一行，保证1-4空格的间隔。 例：

const int A_CONST = 100;         /* 相关的同类注释，可以考虑上下对齐 */
const int ANOTHER_CONST = 200;   /* 上下对齐时，与左侧代码保持间隔 */

当右置的注释超过行宽时，请考虑将注释置于代码上方。

规则4.4.3 不用的代码段直接删除，不要注释掉

被注释掉的代码，无法被正常维护；当企图恢复使用这段代码时，极有可能引入易被忽略的缺陷。 正确的做法是，不需要的代码直接删除掉。若再需要时，考虑移植或重写这段代码。

这里说的注释掉代码，包括用 /* */ 和 //，还包括 #if 0， #ifdef NEVER_DEFINED 等等。

5 头文件

头文件职责

头文件是模块或文件的对外接口，头文件的设计体现了大部分的系统设计。 头文件中适合放置接口的声明，不适合放置实现（内联函数除外）。对于cpp文件中内部才需要使用的函数、宏、枚举、结构定义等不要放在头文件中。 头文件应当职责单一。头文件过于复杂，依赖过于复杂还是导致编译时间过长的主要原因。

建议5.1.1 每一个.cpp文件应有一个对应的.h文件，用于声明需要对外公开的类与接口

通常情况下，每个.cpp文件都有一个相应的.h，用于放置对外提供的函数声明、宏定义、类型定义等。 如果一个.cpp文件不需要对外公布任何接口，则其就不应当存在。 例外：程序的入口（如main函数所在的文件），单元测试代码，动态库代码。

示例:

// Foo.h

#ifndef FOO_H
#define FOO_H

class Foo {
public:
    Foo();
    void Fun();
   
private:
    int value_;
};

#endif

// Foo.cpp
#include "Foo.h"

namespace { // Good: 对内函数的声明放在.cpp文件的头部，并声明为匿名namespace或者static限制其作用域
    void Bar()
    {
    }
}

...

void Foo::Fun()
{
    Bar();
}

头文件依赖

规则5.2.1 禁止头文件循环依赖

头文件循环依赖，指 a.h 包含 b.h，b.h 包含 c.h，c.h 包含 a.h， 导致任何一个头文件修改，都导致所有包含了a.h/b.h/c.h的代码全部重新编译一遍。 而如果是单向依赖，如a.h包含b.h，b.h包含c.h，而c.h不包含任何头文件，则修改a.h不会导致包含了b.h/c.h的源代码重新编译。

头文件循环依赖直接体现了架构设计上的不合理，可通过优化架构去避免。

规则5.2.2 头文件必须编写#define保护，防止重复包含

为防止头文件被重复包含，所有头文件都应当使用 #define 保护；不要使用 #pragma once

定义包含保护符时，应该遵守如下规则： 1）保护符使用唯一名称； 2）不要在受保护部分的前后放置代码或者注释，文件头注释除外。

示例：假定timer模块的timer.h，其目录为timer/include/timer.h,应按如下方式保护：

#ifndef TIMER_INCLUDE_TIMER_H
#define TIMER_INCLUDE_TIMER_H
...
#endif

规则5.2.3 禁止通过声明的方式引用外部函数接口、变量

只能通过包含头文件的方式使用其他模块或文件提供的接口。 通过 extern 声明的方式使用外部函数接口、变量，容易在外部接口改变时可能导致声明和定义不一致。 同时这种隐式依赖，容易导致架构腐化。

不符合规范的案例：

// a.cpp内容

extern int Fun();   // Bad: 通过extern的方式使用外部函数

void Bar()
{
    int i = Fun();
    ...
}

// b.cpp内容

int Fun()
{
    // Do something
}

应该改为：

// a.cpp内容

#include "b.h"   // Good: 通过包含头文件的方式使用其他.cpp提供的接口

void Bar()
{
    int i = Fun();
    ...
}

// b.h内容

int Fun();

// b.cpp内容

int Fun()
{
    // Do something
}

例外，有些场景需要引用其内部函数，但并不想侵入代码时，可以 extern 声明方式引用。 如： 针对某一内部函数进行单元测试时，可以通过 extern 声明来引用被测函数； 当需要对某一函数进行打桩、打补丁处理时，允许 extern 声明该函数。

规则5.2.4 禁止在extern “C"中包含头文件

在 extern “C” 中包含头文件，有可能会导致 extern “C” 嵌套，部分编译器对 extern “C” 嵌套层次有限制，嵌套层次太多会编译错误。

在C，C++混合编程的情况下，在extern “C"中包含头文件，可能会导致被包含头文件的原有意图遭到破坏，比如链接规范被不正确地更改。

示例，存在a.h和b.h两个头文件：

// a.h内容

...
#ifdef __cplusplus
void Foo(int);
#define A(value) Foo(value)
#else
void A(int)
#endif

// b.h内容

...
#ifdef __cplusplus
extern "C" {
#endif

#include "a.h"
void B();

#ifdef __cplusplus
}
#endif

使用C++预处理器展开b.h，将会得到

extern "C" {
    void Foo(int);
    void B();
}

按照 a.h 作者的本意，函数 Foo 是一个 C++ 自由函数，其链接规范为 “C++"。 但在 b.h 中，由于 #include "a.h" 被放到了 extern "C" 的内部，函数 Foo 的链接规范被不正确地更改了。

例外： 如果在 C++ 编译环境中，想引用纯C的头文件，这些C头文件并没有 extern "C" 修饰。非侵入式的做法是，在 extern "C" 中去包含C头文件。

建议5.2.1尽量避免使用前置声明，而是通过#include来包含头文件

前置声明（forward declaration）通常指类、模板的纯粹声明，没伴随着其定义。

  • 优点：
    1. 前置声明能够节省编译时间，多余的 #include 会迫使编译器展开更多的文件，处理更多的输入。
    2. 前置声明能够节省不必要的重新编译的时间。 #include 使代码因为头文件中无关的改动而被重新编译多次。
  • 缺点：
    1. 前置声明隐藏了依赖关系，头文件改动时，用户的代码会跳过必要的重新编译过程。
    2. 前置声明可能会被库的后续更改所破坏。前置声明模板有时会妨碍头文件开发者变动其 API. 例如扩大形参类型，加个自带默认参数的模板形参等等。
    3. 前置声明来自命名空间 std:: 的 symbol 时，其行为未定义（在C++11标准规范中明确说明）。
    4. 前置声明了不少来自头文件的 symbol 时，就会比单单一行的 include 冗长。
    5. 仅仅为了能前置声明而重构代码（比如用指针成员代替对象成员）会使代码变得更慢更复杂。
    6. 很难判断什么时候该用前置声明，什么时候该用#include，某些场景下面前置声明和#include互换以后会导致意想不到的结果。

所以我们尽可能避免使用前置声明，而是使用#include头文件来保证依赖关系。

6 作用域

命名空间

建议6.1.1 对于cpp文件中不需要导出的变量，常量或者函数，请使用匿名namespace封装或者用static修饰

在C++ 2003标准规范中，使用static修饰文件作用域的变量，函数等被标记为deprecated特性，所以更推荐使用匿名namespace。

主要原因如下：

  1. static在C++中已经赋予了太多的含义，静态函数成员变量，静态成员函数，静态全局变量，静态函数局部变量，每一种都有特殊的处理。
  2. static只能保证变量，常量和函数的文件作用域，但是namespace还可以封装类型等。
  3. 统一namespace来处理C++的作用域，而不需要同时使用static和namespace来管理。
  4. static修饰的函数不能用来实例化模板，而匿名namespace可以。

但是不要在 .h 中使用中使用匿名namespace或者static。

// Foo.cpp

namespace {
    const int MAX_COUNT = 20;
    void InternalFun() {};
}

void Foo::Fun()
{
    int i = MAX_COUNT;
   
    InternalFun();
}

规则6.1.1 不要在头文件中或者#include之前使用using导入命名空间

说明：使用using导入命名空间会影响后续代码，易造成符号冲突，所以不要在头文件以及源文件中的#include之前使用using导入命名空间。 示例：

// 头文件a.h
namespace NamespaceA {
    int Fun(int);
}

// 头文件b.h
namespace NamespaceB {
    int Fun(int);
}

using namespace NamespaceB;

void G()
{
    Fun(1);
}

// 源代码a.cpp
#include "a.h"
using namespace NamespaceA;
#include "b.h"

void main()
{
    G(); // using namespace NamespaceA在#include “b.h”之前，引发歧义：NamespaceA::Fun，NamespaceB::Fun调用不明确
}

对于在头文件中使用using导入单个符号或定义别名，允许在模块自定义名字空间中使用，但禁止在全局名字空间中使用。

// foo.h

#include <fancy/string>
using fancy::string;  // Bad，禁止向全局名字空间导入符号

namespace Foo {
    using fancy::string;  // Good，可以在模块自定义名字空间中导入符号
    using MyVector = fancy::vector<int>;  // Good，C++11可在自定义名字空间中定义别名
}

全局函数和静态成员函数

建议6.2.1 优先使用命名空间来管理全局函数，如果和某个class有直接关系的，可以使用静态成员函数

说明：非成员函数放在名字空间内可避免污染全局作用域， 也不要用类+静态成员方法来简单管理全局函数。 如果某个全局函数和某个类有紧密联系， 那么可以作为类的静态成员函数。

如果你需要定义一些全局函数，给某个cpp文件使用，那么请使用匿名namespace来管理。

namespace MyNamespace {
    int Add(int a, int b);
}

class File {
public:
    static File CreateTempFile(const std::string& fileName);
};

全局常量和静态成员常量

建议6.3.1 优先使用命名空间来管理全局常量，如果和某个class有直接关系的，可以使用静态成员常量

说明：全局常量放在命名空间内可避免污染全局作用域， 也不要用类+静态成员常量来简单管理全局常量。 如果某个全局常量和某个类有紧密联系， 那么可以作为类的静态成员常量。

如果你需要定义一些全局常量，只给某个cpp文件使用，那么请使用匿名namespace来管理。

namespace MyNamespace {
    const int MAX_SIZE = 100;
}

class File {
public:
    static const std::string SEPARATOR;
};

全局变量

建议6.4.1 尽量避免使用全局变量，考虑使用单例模式

说明：全局变量是可以修改和读取的，那么这样会导致业务代码和这个全局变量产生数据耦合。

int g_counter = 0;

// a.cpp
g_counter++;

// b.cpp
g_counter++;

// c.cpp
cout << g_counter << endl;

使用单实例模式

class Counter {
public:
    static Counter& GetInstance()
    {
        static Counter counter;
        return counter;
    }  // 单实例实现简单举例
   
    void Increase()
    {
        value_++;
    }
   
    void Print() const
    {
        std::cout << value_ << std::endl;
    }

private:
    Counter() : value_(0) {}

private:
    int value_;
};

// a.cpp
Counter::GetInstance().Increase();

// b.cpp
Counter::GetInstance().Increase();

// c.cpp
Counter::GetInstance().Print();

实现单例模式以后，实现了全局唯一一个实例，和全局变量同样的效果，并且单实例提供了更好的封装性。

例外：有的时候全局变量的作用域仅仅是模块内部，这样进程空间里面就会有多个全局变量实例，每个模块持有一份，这种场景下是无法使用单例模式解决的。

7 类

构造，拷贝构造，赋值和析构函数

构造，拷贝，移动和析构函数提供了对象的生命周期管理方法：

  • 构造函数（constructor）： X()
  • 拷贝构造函数（copy constructor）：X(const X&)
  • 拷贝赋值操作符（copy assignment）：operator=(const X&)
  • 移动构造函数（move constructor）：X(X&&) C++11以后提供
  • 移动赋值操作符（move assignment）：operator=(X&&) C++11以后提供
  • 析构函数（destructor）：~X()

规则7.1.1 类的成员变量必须显式初始化

说明：如果类有成员变量，没有定义构造函数，又没有定义默认构造函数，编译器将自动生成一个构造函数，但编译器生成的构造函数并不会对成员变量进行初始化，对象状态处于一种不确定性。

例外：

    return 0;

Herencia múltiple

En el desarrollo práctico, hay pocos escenarios para usar herencia múltiple, porque la herencia múltiple tiene los siguientes problemas típicos durante su uso:

  1. La herencia en forma de diamante trae duplicación de datos y ambigüedad de nombres. Por lo tanto, C++ introduce la herencia virtual para resolver este tipo de problemas.
  2. Incluso si no es una herencia en forma de diamante, los nombres de múltiples clases base pueden entrar en conflicto, causando ambigüedad.
  3. Cuando una subclase necesita extender o reescribir métodos de múltiples clases base, esto hace que la responsabilidad de la subclase sea poco clara y la semántica caótica.
  4. En comparación con la delegación, la herencia es un tipo de reutilización “white box”, es decir, la subclase puede acceder a los miembros protected de la clase base, lo que genera un acoplamiento más fuerte. Y la herencia múltiple, debido al acoplamiento de múltiples clases base, genera una relación de acoplamiento más fuerte en comparación con la herencia única.

La herencia múltiple tiene las siguientes ventajas: La herencia múltiple proporciona una forma más simple de combinación para lograr la composición y reutilización de múltiples interfaces o clases.

Por lo tanto, solo se permite el uso de herencia múltiple en los siguientes escenarios:

Recomendación 7.3.1 Usar herencia múltiple para implementar separación de interfaces y combinación de múltiples roles

Si una clase necesita implementar múltiples interfaces, se puede usar herencia múltiple para combinar múltiples interfaces separadas, similar a los traits en el lenguaje scala.

    // ...
};

Sobrecarga

La sobrecarga de operadores debe tener una razón suficiente y no debe cambiar la semántica original del operador, por ejemplo, no usar el operador ‘+’ para hacer operaciones de resta. La sobrecarga de operadores hace que el código sea más intuitivo, pero también tiene algunas desventajas:

  • Confunde la intuición, hace creer que la operación es tan eficiente como los tipos integrados, ignorando la posible disminución de rendimiento;
  • La localización de problemas no es tan intuitiva, buscar por nombre de función es obviamente más conveniente que buscar por operador.
  • Si el comportamiento de la sobrecarga del operador no es intuitivo (por ejemplo, usar el operador ‘+’ para hacer operaciones de resta), el código puede generar confusión.
  • La sobrecarga del operador de asignación introduce conversiones implícitas que pueden ocultar bugs muy profundos. Se pueden definir funciones como Equals(), CopyFrom() para reemplazar los operadores =, ==.

8 Funciones

Diseño de funciones

Regla 8.1.1 Evitar funciones demasiado largas, la función no debe superar las 50 líneas (no vacías ni comentarios)

Las funciones deben poder mostrarse en una pantalla (menos de 50 líneas), hacer solo una cosa y hacerlo bien.

Las funciones demasiado largas a menudo significan que la función no es单一, demasiado compleja o demasiado detallada, sin un mayor nivel de abstracción.

Excepción: Algunas funciones que implementan algoritmos, debido a la agregación del algoritmo y la integralidad de la funcionalidad, pueden superar las 50 líneas.

Incluso si una función larga funciona muy bien ahora, una vez que alguien la modifique, puede aparecer un nuevo problema, e incluso causar bugs difíciles de descubrir. Se recomienda dividirla en varias funciones más cortas y manejables para facilitar que otros lean y modifiquen el código.

Funciones en línea

Recomendación 8.2.1 Las funciones en línea no deben superar las 10 líneas (no vacías ni comentarios)

Explicación: Las funciones en línea tienen las características generales de las funciones, lo que las diferencia de las funciones generales es solo el tratamiento de la llamada a la función. Cuando se llama a una función general, el control del programa se transfiere a la función llamada y luego vuelve a la función que la llamó; mientras que al llamar a una función en línea, la expresión de llamada se reemplaza por el cuerpo de la función en línea.

Las funciones en línea son adecuadas solo para funciones pequeñas de 1 a 10 líneas. Para una función grande con muchas sentencias, el costo de llamada y retorno de la función es relativamente insignificante en comparación, y no hay necesidad de usar una función en línea para implementar, generalmente el compilador la tratará como una función ordinaria. Si una función en línea contiene estructuras de control complejas, como bucles, ramificaciones (switch), sentencias try-catch, etc., generalmente el compilador tratará la función como una función ordinaria. Las funciones virtuales y recursivas no pueden usarse como funciones en línea.

Parámetros de función

Recomendación 8.3.1 Usar referencias en lugar de punteros para los parámetros de función

Explicación: Las referencias son más seguras que los punteros, porque siempre son no nulas y nunca pueden apuntar a otro objetivo; las referencias no requieren verificar punteros NULL ilegales.

Si el parámetro de entrada es una matriz cuya longitud es desconocida en tiempo de compilación, se puede usar un puntero en lugar de una referencia. Elija const para evitar que el parámetro sea modificado, permitiendo que el lector del código sepa claramente que el parámetro no será modificado, lo que mejora enormemente la legibilidad del código.

Excepción: Cuando el parámetro de entrada es una matriz cuya longitud es desconocida en tiempo de compilación, se puede usar un puntero en lugar de una referencia.

Recomendación 8.3.2 Usar parámetros de tipo fuerte, evitar usar void*

Aunque diferentes lenguajes tienen sus propias opiniones sobre tipos fuertes y débiles, generalmente se considera que c/c++ es un lenguaje de tipos fuertes. Dado que el lenguaje que usamos es fuertemente tipado, deberíamos mantener este estilo. El beneficio es permitir que el compilador detecte problemas de incompatibilidad de tipos en la fase de compilación.

El uso de tipos fuertes ayuda al compilador a descubrir errores, como en el siguiente código se presta atención al uso de FooListAddNode:

Recomendación 8.3.3 El número de parámetros de la función no debe superar los 5

Demasiados parámetros en una función hacen que la función sea más susceptible a cambios externos, afectando el trabajo de mantenimiento. Además, demasiados parámetros también aumentan la carga de trabajo de pruebas.

  • Ver si se pueden combinar los parámetros relacionados, definiendo una estructura

Constantes e inicialización

Los valores inmutables son más fáciles de entender, rastrear y analizar, por lo que se debe usar constantes en lugar de variables siempre que sea posible, y const debe ser la opción predeterminada al definir valores.

Regla 9.1.1 No está permitido usar macros para representar constantes

Explicación: Las macros son reemplazos de texto simples, completados en la fase de preprocesamiento, errores en tiempo de ejecución se reportan directamente con el valor; durante el seguimiento de depuración también se muestra el valor, no el nombre de la macro; las macros no tienen verificación de tipos, no son seguras; las macros no tienen alcance.

#define MAX_MSISDN_LEN 20    // No es bueno

// Use const constant para C++

// Para versiones C++11 y posteriores, se puede usar constexpr

Regla 9.1.2 No está permitido usar números mágicos

Los llamados números mágicos son números que no se pueden entender, difíciles de comprender.

Los números mágicos no son un concepto de blanco o negro, la dificultad de comprensión también tiene grados, debe juzgarse por uno mismo. Por ejemplo, el número 12, en diferentes contextos, la situación es diferente: type = 12; no se entiende, pero monthsCount = yearsCount * 12; se entiende. El número 0 a veces también es un número mágico, como status = 0; no puede expresar qué estado es.

Vías de solución: Para números locales, se puede añadir una anotación explicativa. Para números utilizados en múltiples lugares, se debe definir const constante, y autoanotar a través del nombre del símbolo.

Prohibido aparecer las siguientes situaciones: No explicar el significado del número a través del símbolo, como const int ZERO = 0 El nombre del símbolo limita su valor, como const int XX_TIMER_INTERVAL_300MS = 300, usar directamente XX_TIMER_INTERVAL_MS para indicar que esta constante es el intervalo de tiempo del temporizador.

Explicación: Una constante solo debe representar una función específica, es decir, una constante no puede tener múltiples usos.

const unsigned int B_MAX_MSISDN_LEN = 20;

// O usar diferentes espacios de nombres:
namespace Namespace1 {
    const unsigned int MAX_MSISDN_LEN = 20;
}

namespace Namespace2 {
    const unsigned int MAX_MSISDN_LEN = 20;
}

Regla 9.1.4 Prohibido usar memcpy_s, memset_s para inicializar objetos no POD

Explicación: POD es el acrónimo de Plain Old Data, es un concepto introducido en el estándar C++ 98 (ISO/IEC 14882, primera edición, 1998-09-01), los tipos POD incluyen principalmente tipos primitivos como int, char, float, double, enumeration, void, punteros, etc., así como tipos agregados, no pueden usar encapsulamiento y características orientadas a objetos (como constructores/operadores de asignación/destructores definidos por el usuario, clases base, funciones virtuales, etc.).

Dado que los objetos no POD como los objetos de clase tipo no agregado pueden tener funciones virtuales, el diseño de memoria no es determinista, depende del compilador, el uso abusivo de la copia de memoria puede causar graves problemas.

Incluso para los objetos de clase tipo agregado, el uso de operaciones de copia y comparación de memoria directa rompe el efecto de ocultación de información y protección de datos, y tampoco se recomiendan las operaciones memcpy_s, memset_s.

Para una explicación detallada de los tipos POD, consulte el apéndice.

Recomendación 9.1.2 Declare e inicialice las variables cuando se usen

Explicación: No inicializar una variable antes de usarla es un error de programación de bajo nivel común. Declarar una variable cuando se usa y al mismo tiempo inicializarla evita fácilmente este tipo de error de bajo nivel.

Declarar todas las variables al comienzo de la función y usarlas más tarde, su alcance cubre toda la implementación de la función, hace que el programa sea difícil de entender y mantener: la definición de la variable está separada de su uso. Difícil inicializar razonablemente una variable: al comienzo de la función, a menudo no hay suficiente información para inicializar una variable, a menudo se usa un valor vacío predeterminado (como cero) para inicializar, esto a menudo es una pérdida, y si se usa una variable antes de que se le asigne un valor válido, causará errores.

Siguiendo el principio de minimizar el alcance de la variable y el principio de declaración cercana, hace que el código sea más fácil de leer, facilita entender el tipo e inicialización de la variable. Especialmente, se debe usar la inicialización en lugar de la declaración seguida de la asignación.

// Mal ejemplo: Declaración e inicialización separadas
string name;        // No se inicializó al declarar: llamada al constructor por defecto
name = "zhangsan";  // Llamada al operador de asignación nuevamente; declaración y definición en diferentes lugares, difícil de entender

// Buen ejemplo: Declaración e inicialización integradas, más fáciles de entender
string name("zhangsan");  // Llamada al constructor

Expresiones

Regla 9.2.1 Prohibido volver a usar la misma variable en expresiones que contengan operaciones de autoincremento o autodecremento

En expresiones que contengan operaciones de autoincremento o autodecremento de variables, si se vuelve a usar esa variable, el resultado no está claramente definido en el estándar C++. Diferentes compiladores o diferentes versiones del mismo compilador pueden tener implementaciones diferentes. Para una mayor portabilidad, no se debe asumir ningún orden de operación no definido por el estándar.

Tenga en cuenta que los problemas de orden de operación no pueden resolverse usando paréntesis, porque esto no es un problema de prioridad.

Ejemplo:

x = b[i] + i++; // Bad: El orden de b[i] y i++ no está claro.

La forma correcta de escribirlo es colocar la operación de autoincremento o autodecremento en una línea separada:

x = b[i] + i;
i++;            // Good: En una línea separada

Parámetros de función

Func(i++, i);   // Bad: Al pasar el segundo parámetro, no está claro si la operación de autoincremento ha ocurrido

Forma correcta de escribirlo

i++;            // Good: En una línea separada
x = Func(i, i);

Regla 9.2.2 Las sentencias switch deben tener una rama default

En la mayoría de los casos, las sentencias switch deben tener una rama default para garantizar un tratamiento predeterminado en caso de que se omita alguna etiqueta case. Esto proporciona un tratamiento predeterminado en caso de omisiones.

Excepción: Si la variable de condición switch es un tipo enumerado y las ramas case cubren todos los valores, agregar una rama default puede ser innecesario. Los compiladores modernos tienen la capacidad de verificar si se omiten algunos valores de enumeración en las ramas case de una sentencia switch, y habrá una advertencia correspondiente.

enum Color {
    RED = 0,
    BLUE
};

// Como la variable de condición switch es un enumerado, aquí no es necesario agregar una rama default
switch (color) {
    case RED:
        DoRedThing();
        break;
    case BLUE:
        DoBlueThing();
        ...
        break;
}

Recomendación 9.2.1 La comparación de expresiones debe seguir el principio de que el lado izquierdo tiende a cambiar y el lado derecho tiende a permanecer constante.

Cuando se compara una variable con una constante, si la constante está a la izquierda, como if (MAX == v) no se ajusta al hábito de lectura, mientras que if (MAX > v) es aún más difícil de entender. Se debe seguir el hábito de lectura y expresión humanos, poniendo la constante a la derecha. Escrito de la siguiente manera:

if (value == MAX) {
 
}

if (value < MAX) {
 
}

Hay excepciones, como if (MIN < value && value < MAX) cuando se describe un rango, la primera mitad tiene la constante a la izquierda.

No se debe preocupar por confundir ‘==’ con ‘=’, porque if (value = MAX) tendrá una advertencia de compilación, y otras herramientas de inspección estática también reportarán errores. Deje que las herramientas resuelvan problemas de escritura, el código debe cumplir con la primera legibilidad.

Recomendación 9.2.2 Use paréntesis para aclarar la precedencia de operadores

Use paréntesis para aclarar la precedencia de operadores, evitando que la precedencia predeterminada no coincida con el diseño intencional y cause errores en el programa; al mismo tiempo, hace que el código sea más claro y legible. Sin embargo, demasiados paréntesis dispersan el código y reducen la legibilidad. A continuación se presentan sugerencias sobre cómo usar paréntesis.

  • Operadores binarios o de mayor grado, si hay varios tipos de operadores, use paréntesis.
x = a + b + c;         /* Operadores iguales, no se necesitan paréntesis */
x = Foo(a + b, c);     /* Expresiones separadas por comas, no se necesitan paréntesis */
x = 1 << (2 + 3);      /* Operadores diferentes, se necesitan paréntesis */
x = a + (b / 5);       /* Operadores diferentes, se necesitan paréntesis */
x = (a == b) ? a : (a – b);    /* Operadores diferentes, se necesitan paréntesis */

Conversión de tipos

Evite usar ramificaciones de tipos para personalizar el comportamiento: el uso de ramificaciones de tipos para personalizar el comportamiento es propenso a errores y es una señal clara de intentar escribir código C con C++. Esta es una técnica inflexible, al agregar un nuevo tipo, si se olvida de modificar todas las ramas, el compilador tampoco lo notificará. Use plantillas y funciones virtuales para que el tipo en sí, en lugar del código que las llama, decida el comportamiento.

Se recomienda evitar la conversión de tipos, debemos considerar el tipo de datos de cada dato al diseñar el tipo de datos en nuestro código, en lugar de depender excesivamente de la conversión de tipos para resolver problemas. Al diseñar un tipo básico, considere:

  • Si es con signo o sin signo
  • Si es adecuado para float o double
  • Si es adecuado para int8, int16, int32 o int64, determine la longitud del tipo entero

Sin embargo, no podemos prohibir el uso de la conversión de tipos, porque el lenguaje C++ es un lenguaje de programación orientado a la máquina, involucra punteros y direcciones, y nos comunicamos con diversas API de terceros o de bajo nivel, cuyo diseño de tipos puede no ser razonable, en este proceso de adaptación es fácil encontrar conversiones de tipos.

Excepción: Cuando se llama a una función, si no se desea manejar el resultado de la función, primero considere si esta es su mejor opción. Si realmente no desea manejar el valor de retorno de la función, puede usar una conversión (void) para resolverlo.

Regla 9.3.1 Si se determina usar conversión de tipos, use la conversión proporcionada por C++ en lugar de la conversión de estilo C

Explicación:

La conversión proporcionada por C++ es más específica que la conversión de estilo C, es más legible y más segura. C++ proporciona las siguientes conversiones:

  • Conversión de tipo:
  1. dynamic_cast: Principalmente utilizado para conversiones descendentes en la jerarquía de herencia, dynamic_cast tiene la función de verificación de tipo, por favor diseñe bien la clase base y la clase derivada, y evite usar dynamic_cast para la conversión.
  2. static_cast: Similar a la conversión de estilo C, puede hacer conversiones forzadas de valores o conversiones ascendentes (convertir un puntero o referencia de clase derivada en un puntero o referencia de clase base). Esta conversión a menudo se usa para eliminar la ambigüedad causada por la herencia múltiple, es relativamente segura. Si es una conversión aritmética pura, se recomienda usar el método de inicialización con llaves a continuación.
  3. reinterpret_cast: Se utiliza para convertir tipos no relacionados. reinterpret_cast obliga al compilador a reinterpretar la memoria de un tipo de objeto como otro tipo, esto es inseguro, se recomienda usar reinterpret_cast lo menos posible.
  4. const_cast: Se utiliza para eliminar la propiedad const de un objeto, haciendo que el objeto sea modificable, esto rompe la inmutabilidad de los datos, se recomienda usarlo lo menos posible.
  • Conversión aritmética: (soportada desde C++11) Para conversiones aritméticas donde no se pierde información de tipo, como de float a double, de int32 a int64, se recomienda usar el método de inicialización con llaves.
  double d{ someFloat };
  int64_t i{ someInt32 };

Recomendación 9.3.1 Evitar usar dynamic_cast

  1. dynamic_cast depende del RTTI de C++, permite a los programadores reconocer el tipo de objeto de la clase C++ en tiempo de ejecución.
  2. El uso de dynamic_cast generalmente indica un problema con el diseño de nuestra clase base y clase derivada, la clase derivada rompe el contrato de la clase base, forzando la conversión a la subclase a través de dynamic_cast para un tratamiento especial, en este momento es mejor mejorar el diseño de la clase en lugar de resolver el problema a través de dynamic_cast.

Recomendación 9.3.2 Evitar usar reinterpret_cast

Explicación: reinterpret_cast se utiliza para convertir tipos no relacionados.

Intentar forzar un tipo a otro tipo a través de reinterpret_cast rompe la seguridad y confiabilidad del tipo, es una conversión insegura. Se debe evitar la conversión entre diferentes tipos.

Recomendación 9.3.3 Evitar usar const_cast

Explicación: const_cast se utiliza para eliminar la propiedad const y volatile de un objeto.

Usar const_cast para modificar un objeto const mediante el puntero o referencia convertido tiene un comportamiento indefinido.

// Mal ejemplo
const int i = 1024;
int* p = const_cast<int*>(&i);
*p = 2048;      // Comportamiento indefinido

// Mal ejemplo
class Foo {
public:
    Foo() : i(3) {}

    void Fun(int v)
    {
        i = v;
    }

private:
    int i;
};

int main(void)
{
    const Foo f;
    Foo* p = const_cast<Foo*>(&f);
    p->Fun(8);  // Comportamiento indefinido
}

Asignación y liberación de recursos

Regla 9.4.1 Para la liberación de un solo objeto, use delete; para la liberación de un array de objetos, use delete[]

Explicación: Para la liberación de un solo objeto, use delete; para la liberación de un array de objetos, use delete[]. Motivo:

  • La acción de llamar a new incluye: solicitar un bloque de memoria del sistema y llamar al constructor de este tipo.
  • La acción de llamar a new[n] incluye: solicitar memoria capaz de contener n objetos y llamar al constructor de cada objeto.
  • La acción de llamar a delete incluye: primero llamar al destructor correspondiente, luego devolver la memoria al sistema.
  • La acción de llamar a delete[] incluye: llamar al destructor de cada objeto, luego liberar toda la memoria.

Si el formato de new y delete no coincide, el resultado es desconocido. Para tipos que no sean de clase, new y delete no llaman a constructores y destructores.

Código incorrecto:

const int MAX_ARRAY_SIZE = 100;
int* numberArray = new int[MAX_ARRAY_SIZE];
...
delete numberArray;
numberArray = nullptr;

Código correcto:

const int MAX_ARRAY_SIZE = 100;
int* numberArray = new int[MAX_ARRAY_SIZE];
...
delete[] numberArray;
numberArray = nullptr;

Recomendación 9.4.1 Utilizar la característica RAII para ayudar a rastrear la asignación dinámica

Explicación: RAII es el acrónimo de “Resource Acquisition Is Initialization” (La adquisición de recursos es la inicialización), es una técnica simple que utiliza el ciclo de vida del objeto para controlar los recursos del programa (como memoria, manejadores de archivos, conexiones de red, mutexes, etc.).

La práctica general de RAII es la siguiente: adquirir recursos durante la construcción del objeto, luego controlar el acceso a los recursos para que permanezcan válidos durante el ciclo de vida del objeto, y finalmente liberar los recursos durante la destrucción del objeto. Esta práctica tiene dos grandes ventajas:

  • No necesitamos liberar recursos explícitamente.
  • Los recursos requeridos por el objeto permanecen válidos durante todo su ciclo de vida. De esta manera, no es necesario verificar la validez de los recursos, lo que simplifica la lógica y mejora la eficiencia.

Ejemplo: Usar RAII no requiere liberar recursos mutex explícitamente.

class LockGuard {
public:
    LockGuard(const LockType& lockType): lock_(lockType)
    {
        lock_.Acquire();
    }
   
    ~LockGuard()
    {
        lock_.Release();
    }
   
private:
    LockType lock_;
};


bool Update()
{
    LockGuard lockGuard(mutex);
    if (...) {
        return false;
    } else {
        // operar datos
    }
   
    return true;
}

Biblioteca estándar

El grado de uso de STL en diferentes productos varía, aquí se enumeran algunas reglas y recomendaciones básicas para referencia de cada equipo.

Regla 9.5.1 No guardar el puntero devuelto por c_str() de std::string

Explicación: El estándar C++ no especifica que el puntero devuelto por string::c_str() sea persistente, por lo tanto, una implementación específica de STL puede devolver un almacenamiento temporal en string::c_str() y liberarlo rápidamente. Por lo tanto, para garantizar la portabilidad del programa, no debe guardar el resultado de string::c_str(), sino llamar directamente cada vez que se necesite.

Ejemplo:

void Fun1()
{
    std::string name = "demo";
    const char* text = name.c_str();  // Después de que termine la expresión, el ciclo de vida de name sigue existiendo, el puntero es válido

    // Si se llama a una función miembro no const de string en medio, causando que string sea modificado, como operator[], begin() etc.
    // Puede causar que el contenido de text ya no esté disponible, o ya no sea la cadena original
    name = "test";
    name[1] = '2';

    // Uso posterior del puntero text, su contenido ya no es "demo"
}

void Fun2()
{
    std::string name = "demo";
    std::string test = "test";
    const char* text = (name + test).c_str(); // Después de que termine la expresión, el objeto temporal creado por + es destruido, el puntero es inválido

    // Uso posterior del puntero text, ya no apunta a un espacio de memoria válido
}

Excepción: En algunos códigos con requisitos de alto rendimiento, para adaptarse a funciones existentes que solo aceptan parámetros de tipo const char*, se puede guardar temporalmente el puntero devuelto por string::c_str(). Pero debe garantizarse estrictamente que el ciclo de vida del objeto string sea más largo que el ciclo de vida del puntero guardado, y garantizar que durante el ciclo de vida del puntero guardado, el objeto string no sea modificado.

Recomendación 9.5.1 Utilizar std::string en lugar de char*

Explicación: Utilizar string en lugar de char* tiene muchas ventajas, por ejemplo:

  1. No es necesario considerar el carácter ‘\0’ al final;
  2. Se pueden usar directamente operadores como +, =, == y otras funciones de operación de cadenas;
  3. No es necesario considerar la operación de asignación de memoria, evitando new/delete explícitos y los errores que pueden causar;

Es importante tener en cuenta que en algunas implementaciones de STL, string se basa en una estrategia de copia en escritura, lo que trae dos problemas: primero, algunas versiones de la estrategia de copia en escritura no implementan seguridad de subprocesos, lo que puede causar el colapso del programa en un entorno multihilo; segundo, cuando se pasa string basado en copia en escritura entre bibliotecas dinámicas, el contador de referencias no puede reducirse cuando la biblioteca dinámica es descargada, lo que puede causar punteros colgantes. Por lo tanto, elegir una implementación de STL confiable es muy importante para garantizar la estabilidad del programa.

Excepción: Cuando se llama a la API del sistema o de terceros, para interfaces ya definidas, solo se puede usar char*. Pero antes de llamar a la interfaz, se puede usar string, y al llamar a la interfaz, use string::c_str() para obtener el puntero de caracteres. Cuando se asigna un array de caracteres en la pila como buffer, se puede definir directamente el array de caracteres, no se debe usar string, tampoco es necesario usar contenedores como vector<char>.

Regla 9.5.2 Prohibido usar auto_ptr

Explicación: En la biblioteca STL, std::auto_ptr tiene un comportamiento implícito de transferencia de propiedad, como en el siguiente código:

auto_ptr<T> p1(new T);
auto_ptr<T> p2 = p1;

Después de ejecutar la segunda línea, p1 ya no apunta al objeto asignado en la primera línea, sino que se convierte en nullptr. Debido a esto, auto_ptr no puede ser colocado en varios contenedores estándar. El comportamiento de transferencia de propiedad a menudo no es el resultado esperado. Para escenarios donde es necesario transferir la propiedad, tampoco se debe usar un método de transferencia implícita. Esto a menudo requiere que los programadores mantengan una precaución extra con el código que usa auto_ptr, de lo contrario se producirá un acceso a un puntero nulo. Hay dos escenarios comunes para usar auto_ptr, uno es pasar auto_ptr fuera de la función que lo produce, y el otro es usar auto_ptr como una clase RAII para gestionar recursos, liberando los recursos automáticamente cuando el ciclo de vida de auto_ptr termina. Para el primer escenario, se puede usar std::shared_ptr en su lugar. Para el segundo escenario, se puede usar std::unique_ptr del estándar C++11. std::unique_ptr es el reemplazo de std::auto_ptr, soporta transferencia de propiedad explícita.

Excepción: Antes de que el estándar C++11 sea ampliamente utilizado, en escenarios donde es absolutamente necesario transferir la propiedad, se puede usar std::auto_ptr, pero se recomienda encapsular std::auto_ptr y prohibir el constructor de copia y el operador de asignación de la clase encapsulada, para que la clase encapsulada no pueda ser usada en contenedores estándar.

Recomendación 9.5.2 Usar nuevas cabeceras estándar

Explicación: Al usar las cabeceras estándar de C++, use <cstdlib> en lugar de <stdlib.h>.

Uso de const

Al agregar la palabra clave const antes de las variables o parámetros declarados, indica que el valor de la variable no puede ser alterado (por ejemplo, const int foo). Al agregar el calificador const a las funciones de una clase, indica que la función no modificará el estado de las variables miembro de la clase (por ejemplo, class Foo { int Bar(char c) const; };). Las variables, miembros de datos, funciones y parámetros const añaden una capa de protección a la detección de errores en tiempo de compilación, facilitando la detección temprana de errores. Por lo tanto, recomendamos encarecidamente usar const en cualquier caso posible. A veces, usar constexpr de C++11 para definir constantes reales puede ser mejor.

Regla 9.6.1 Para parámetros de puntero y referencia, si no se necesita modificar, use const

Los valores inmutables son más fáciles de entender/seguir y analizar, use const como opción predeterminada, será verificado en tiempo de compilación, haciendo el código más robusto/seguro.

class Foo;

void PrintFoo(const Foo& foo);

Regla 9.6.2 Para funciones miembro que no modifiquen variables miembro, use const para modificar

Declare las funciones miembro como const siempre que sea posible. Las funciones de acceso siempre deben ser const. Todas las funciones miembro que no modifiquen los miembros de datos deben declararse como const. Para funciones virtuales, se debe considerar el propósito del diseño de si todas las clases en la cadena de herencia necesitan modificar miembros de datos en esta función virtual, en lugar de solo considerar la implementación de una sola clase.

class Foo {
public:

    // ...

    int PrintValue() const // const modifica la función miembro, no modificará variables miembro
    {
        std::cout << value_ << std::endl;
    }

    int GetValue() const  // const modifica la función miembro, no modificará variables miembro
    {
        return value_;
    }

private:
    int value_;
};

Excepciones

Recomendación 9.7.1 En C++11, si una función no lanzará excepciones, declare como noexcept

Razón

  1. Si una función no lanzará excepciones, declararla como noexcept permite al compilador optimizar la función al máximo, como reducir las rutas de ejecución y mejorar la eficiencia de la salida de errores.
  2. Contenedores como vector, para garantizar la robustez de la interfaz, si el operador move del elemento almacenado no está declarado como noexcept, al ampliar el contenedor y mover elementos, no usará el mecanismo move, sino el mecanismo copy, lo que conlleva el riesgo de pérdida de rendimiento. Si una función no puede lanzar excepciones, o un programa no captura la excepción lanzada por una función y la maneja, entonces esta función puede usar la nueva palabra clave noexcept para modificarla, indicando que esta función no lanzará excepciones o que las excepciones lanzadas no serán capturadas y manejadas. Por ejemplo:
extern "C" double sqrt(double) noexcept;  // Nunca lanzará excepciones

// Incluso si puede lanzar excepciones, también se puede usar noexcept
// Aquí no se pretende manejar la excepción de agotamiento de memoria, simplemente se declara la función como noexcept
std::vector<int> MyComputation(const std::vector<int>& v) noexcept
{
    std::vector<int> res = v;    // Puede lanzar excepciones
    // hacer algo
    return res;
}

Ejemplo

RetType Function(Type params) noexcept;   // Máxima optimización
RetType Function(Type params);            // Menos optimización

// El move de std::vector necesita declararse como noexcept
class Foo1 {
public:
    Foo1(Foo1&& other);  // no noexcept
};

std::vector<Foo1> a1;
a1.push_back(Foo1());
a1.push_back(Foo1());  // Desencadena la expansión del contenedor, al mover elementos existentes llama al constructor de copia

class Foo2 {
public:
    Foo2(Foo2&& other) noexcept;
};

std::vector<Foo2> a2;
a2.push_back(Foo2());
a2.push_back(Foo2());  // Desencadena la expansión del contenedor, al mover elementos existentes llama al constructor de movimiento

Nota Los constructores por defecto, destructores, funciones swap y operadores de movimiento no deberían lanzar excepciones.

Plantillas y programación genérica

Regla 9.8.1 Prohibido programar genéricamente en el proyecto OpenHarmony

La programación genérica y la programación orientada a objetos tienen ideas, conceptos y técnicas completamente diferentes, el proyecto OpenHarmony utiliza principalmente ideas orientadas a objetos.

C++ proporciona un mecanismo potente de programación genérica, capaz de implementar interfaces seguras para tipos que son flexibles y concisas, permitiendo la reutilización de código con comportamiento idéntico pero tipos diferentes.

Sin embargo, la programación genérica en C++ tiene las siguientes desventajas:

  1. Las personas que no están muy familiarizadas con la programación genérica a menudo escriben lógica orientada a objetos como plantillas, escriben miembros no dependientes de parámetros de plantilla en plantillas, etc., lo que provoca problemas como confusión lógica y expansión de código.
  2. La programación con plantillas es a menudo oscura y difícil de entender para personas que no están muy familiarizadas con C++. El código que usa plantillas en lugares complejos es más difícil de leer, y es complicado depurar y mantener.
  3. La programación con plantillas a menudo genera mensajes de error de compilación poco amigables: cuando el código tiene errores, aunque la interfaz es muy simple, los detalles internos de la plantilla también se muestran en el mensaje de error. Esto hace que el mensaje de error sea muy difícil de entender.
  4. Si las plantillas no se usan adecuadamente, pueden causar una excesiva expansión del código en tiempo de ejecución.
  5. El código de plantilla es difícil de modificar y refactorizar. El código de plantilla se expande en muchos contextos, por lo que es difícil confirmar que la refactorización es útil para todo el código expandido.

Por lo tanto, la mayoría de los componentes de OpenHarmony prohíben la programación con plantillas, solo unos pocos componentes pueden usar programación genérica, y el código de plantilla desarrollado debe tener comentarios detallados. Excepción:

  1. La capa de adaptación de STL puede usar plantillas

Macros

En el lenguaje C++, recomendamos encarecidamente usar macros complejas lo menos posible.

  • Para la definición de constantes, siga lo descrito en los capítulos anteriores, use const o enum;
  • Para macros de funciones, trate de que sean simples y siga los principios a continuación, y priorice el uso de funciones en línea, funciones de plantilla, etc., para reemplazarlas.
// No se recomienda usar macros de funciones
#define SQUARE(a, b) ((a) * (b))

// Por favor use funciones de plantilla, funciones en línea, etc. para reemplazar.
template<typename T> T Square(T a, T b) { return a * b; }

Si necesita usar macros, consulte el capítulo correspondiente de la norma de lenguaje C. Excepción: Algunas aplicaciones generales y maduras, como el encapsulado y manejo de new, delete, pueden conservar el uso de macros.

10 Características modernas de C++

Con la publicación del estándar de lenguaje C++11 por ISO en 2011, y la publicación de C++17 en marzo de 2017, C++ moderno (C++11/14/17, etc.) ha añadido una gran cantidad de características de lenguaje y bibliotecas estándar que mejoran la eficiencia de programación y la calidad del código. Este capítulo describe algunas directrices que pueden ayudar a los equipos a usar C++ moderno de manera más eficiente y evitar trampas del lenguaje.

Mejora de la concisión y seguridad del código

Recomendación 10.1.1 Uso razonable de auto

Razón

  • auto puede evitar escribir nombres de tipos largos y repetitivos, también puede garantizar la inicialización de la variable al definirla.
  • La regla de deducción de tipo de auto es compleja, necesita una comprensión cuidadosa.
  • Si puede hacer que el código sea más claro, continúe usando el tipo explícito, y solo use auto para variables locales.

Ejemplo

// Evita nombres de tipos largos
std::map<string, int>::iterator iter = m.find(val);
auto iter = m.find(val);

// Evita nombres de tipos repetitivos
class Foo {...};
Foo* p = new Foo;
auto p = new Foo;

// Garantiza la inicialización
int x;    // Compila correctamente, no está inicializado
auto x;   // Falla la compilación, debe estar inicializado

La deducción de tipo de auto puede llevar a confusiones:

auto a = 3;           // int
const auto ca = a;    // const int
const auto& ra = a;   // const int&
auto aa = ca;         // int, ignora const y referencia
auto ila1 = { 10 };   // std::initializer_list<int>
auto ila2{ 10 };      // std::initializer_list<int>

auto&& ura1 = x;      // int&
auto&& ura2 = ca;     // const int&
auto&& ura3 = 10;     // int&&

const int b[10];
auto arr1 = b;        // const int*
auto& arr2 = b;       // const int(&)[10]

Si no se presta atención a que la deducción de tipo auto ignora la referencia, puede introducir problemas de rendimiento difíciles de detectar:

std::vector<std::string> v;
auto s1 = v[0];  // auto se deduce como std::string, copia v[0]

Si se usa auto para definir interfaces, como constantes en archivos de encabezado, puede cambiar el tipo debido a que los desarrolladores modifican el valor.

Regla 10.1.1 Al sobrescribir funciones virtuales, use la palabra clave override o final

Razón Las palabras clave override y final pueden garantizar que la función es virtual y sobrescribe la función virtual de la clase base. Si el prototipo de la función de la subclase no coincide con el de la clase base, generará una advertencia de compilación. final también garantiza que la función virtual no será sobrescrita por la subclase.

Usar la palabra clave override o final permite detectar errores en tiempo de compilación si se modifica el prototipo de la función virtual de la clase base pero se olvida de modificar la función virtual sobrescrita en la subclase. También puede evitar la omisión de la modificación de funciones virtuales sobrescritas cuando hay múltiples subclases.

Ejemplo

class Base {
public:
    virtual void Foo();
    virtual void Foo(int var);
    void Bar();
};

class Derived : public Base {
public:
    void Foo() const override; // Fallo de compilación: Derived::Foo y Base::Foo no coinciden en el prototipo, no es una sobrescritura
    void Foo() override;       // Correcto: Derived::Foo sobrescribe Base::Foo
    void Foo(int var) final;   // Correcto: Derived::Foo(int) sobrescribe Base::Foo(int), y las subclases de Derived ya no pueden sobrescribir esta función
    void Bar() override;       // Fallo de compilación: Base::Bar no es una función virtual
};

Resumen

  1. La clase base define la función virtual por primera vez, use la palabra clave virtual
  2. La subclase sobrescribe la función virtual de la clase base (incluyendo el destructor), use la palabra clave override o final (pero no ambas), y no use la palabra clave virtual
  3. Para funciones no virtuales, no use virtual, override ni final

Regla 10.1.2 Use la palabra clave delete para eliminar funciones

Razón En comparación con declarar funciones miembro de clase como private pero no implementarlas, la palabra clave delete es más explícita y tiene un alcance de aplicación más amplio.

Ejemplo

class Foo {
private:
    // No se sabe al ver el archivo de encabezado si el constructor de copia está eliminado
    Foo(const Foo&);
};

class Foo {
public:
    // Elimina explícitamente la función de asignación de copia
    Foo& operator=(const Foo&) = delete;
};

La palabra clave delete también admite la eliminación de funciones no miembro

template<typename T>
void Process(T value);

template<>
void Process<void>(void) = delete;

Regla 10.1.3 Use nullptr, no NULL ni 0

Razón Durante mucho tiempo, C++ no tenía una palabra clave para representar un puntero nulo, lo cual era incómodo:

#define NULL ((void *)0)

char* str = NULL;   // Error: void* no se puede convertir automáticamente a char*

void(C::*pmf)() = &C::Func;
if (pmf == NULL) {} // Error: void* no se puede convertir automáticamente a puntero a función de miembro

Si NULL se define como 0 o 0L. Puede resolver el problema anterior.

O se puede usar directamente 0 donde se necesite un puntero nulo. Pero esto introduce otro problema, el código no es claro, especialmente al usar la deducción automática auto:

auto result = Find(id);
if (result == 0) {  // Find() devuelve un puntero o un entero?
    // hacer algo
}

0 es literalmente de tipo int (0L es de tipo long), por lo que ni NULL ni 0 son tipos de puntero. Cuando se sobrecargan funciones de puntero y entero, pasar NULL o 0 llama a la función sobrecargada de tipo entero:

void F(int);
void F(int*);

F(0);      // Llama a F(int), no a F(int*)
F(NULL);   // Llama a F(int), no a F(int*)

Además, sizeof(NULL) == sizeof(void*) no siempre es cierto, esto también es un riesgo potencial.

Resumen: Usar directamente 0 o 0L hace que el código no sea claro y no sea seguro para tipos; usar NULL no es seguro para tipos. Estos son riesgos potenciales.

La ventaja de nullptr no es solo que representa literalmente un puntero nulo, haciendo el código claro, sino que ya no es un tipo entero.

nullptr es de tipo std::nullptr_t, y std::nullptr_t se puede convertir implícitamente a todos los tipos de puntero primitivo, lo que hace que nullptr se comporte como un puntero nulo que apunta a cualquier tipo.

void F(int);
void F(int*);
F(nullptr);   // Llama a F(int*)

auto result = Find(id);
if (result == nullptr) {  // Find() devuelve un puntero
    // hacer algo
}

Regla 10.1.4 Use using en lugar de typedef

Antes de C++11, se podía definir alias de tipo mediante typedef. Nadie quiere repetir std::map<uint32_t, std::vector<int>> muchas veces.

typedef std::map<uint32_t, std::vector<int>> SomeType;

Los alias de tipo son en realidad encapsulamiento de tipos. Y a través del encapsulamiento, el código puede ser más claro, y en gran medida evitar la modificación de tipo de cambio de escopeta. Después de C++11, se proporciona using, implementando declaraciones de alias(alias declarations):

using SomeType = std::map<uint32_t, std::vector<int>>;

Comparando los formatos de ambos:

typedef Type Alias;   // Type primero o Alias primero
using Alias = Type;   // Cumple con el uso de 'asignación', es fácil de entender y no propenso a errores

Si siente que esto aún no es suficiente para cambiar a using, veamos alias template(plantilla de alias):

// Definición de alias de plantilla, una línea de código
template<class T>
using MyAllocatorVector = std::vector<T, MyAllocator<T>>;

MyAllocatorVector<int> data;       // Uso del alias definido con using

template<class T>
class MyClass {
private:
    MyAllocatorVector<int> data_;   // Uso del alias definido con using en una clase de plantilla
};

Y typedef no admite alias con parámetros de plantilla, solo se puede usar un “camino curvo”:

// A través de la plantilla que envuelve typedef, se necesita implementar una clase de plantilla
template<class T>
struct MyAllocatorVector {
    typedef std::vector<T, MyAllocator<T>> type;
};

MyAllocatorVector<int>::type data;  // Uso del alias definido con typedef, se necesita escribir ::type

template<class T>
class MyClass {
private:
    typename MyAllocatorVector<int>::type data_;  // Uso en una clase de plantilla, además de ::type, también se necesita typename
};

Regla 10.1.5 Prohibido usar std::move en objetos const

A primera vista, std::move significa mover un objeto. Y un objeto const no se puede modificar, mucho menos mover. Por lo tanto, usar std::move en un objeto const puede confundir al lector del código. En cuanto a la funcionalidad real, std::move convertirá el objeto en un tipo de referencia rvalue; para un objeto const, lo convertirá en una referencia rvalue const. Dado que rara vez hay tipos que definan constructores de movimiento y operadores de asignación de movimiento con parámetros de referencia rvalue const, la funcionalidad del código a menudo se reduce a la copia del objeto en lugar de moverlo, lo que trae una pérdida de rendimiento.

Ejemplo incorrecto:

std::string g_string;
std::vector<std::string> g_stringList;

void func()
{
    const std::string myString = "String content";
    g_string = std::move(myString); // malo: no mueve myString, sino que realiza una copia
    const std::string anotherString = "Another string content";
    g_stringList.push_back(std::move(anotherString));    // malo: no mueve anotherString, sino que realiza una copia
}

Punteros inteligentes

Regla 10.2.1 Para objetos de单例, miembros de clase, etc., que no comparten propiedad, use punteros crudos en lugar de punteros inteligentes

Razón Los punteros inteligentes liberan automáticamente los recursos del objeto para evitar fugas de recursos, pero traen gastos generales adicionales. Por ejemplo: clases generadas automáticamente por punteros inteligentes, gastos de construcción y destrucción, más uso de memoria, etc.

Para objetos como单例, miembros de clase, etc., donde la propiedad no es compartida por múltiples partes, solo es necesario liberar recursos en la destrucción de la clase. No se debe usar punteros inteligentes para aumentar gastos innecesarios.

Ejemplo

class Foo;
class Base {
public:
    Base() {}
    virtual ~Base()
    {
        delete foo_;
    }
private:
    Foo* foo_ = nullptr;
};

Excepción

  1. Al devolver objetos creados, cuando se necesita una función de destrucción de puntero, se puede usar un puntero inteligente.
class User;
class Foo {
public:
    std::unique_ptr<User, void(User *)> CreateUniqueUser() // se puede usar unique_ptr para garantizar la creación y liberación del objeto en el mismo runtime
    {
        sptr<User> ipcUser = iface_cast<User>(remoter);
        return std::unique_ptr<User, void(User *)>(::new User(ipcUser), [](User *user) {
            user->Close();
            ::delete user;
        });
    }

    std::shared_ptr<User> CreateSharedUser() // se puede usar shared_ptr para garantizar la creación y liberación del objeto en el mismo runtime
    {
        sptr<User> ipcUser = iface_cast<User>(remoter);
        return std::shared_ptr<User>(ipcUser.GetRefPtr(), [ipcUser](User *user) mutable {
            ipcUser = nullptr;
        });
    }
};
  1. Cuando se devuelven objetos creados y el objeto necesita ser referenciado por múltiples partes, se puede usar shared_ptr.

Regla 10.2.2 Use std::make_unique en lugar de new para crear unique_ptr

Razón

  1. make_unique proporciona una forma más concisa de creación.
  2. Garantiza la seguridad de excepciones en expresiones complejas.

Ejemplo

// No es bueno: MyClass aparece dos veces, riesgo de inconsistencia por repetición
std::unique_ptr<MyClass> ptr(new MyClass(0, 1));
// Bueno: MyClass aparece una vez, no hay posibilidad de inconsistencia
auto ptr = std::make_unique<MyClass>(0, 1);

La repetición de tipos puede causar problemas muy graves y difíciles de detectar:

// Compila correctamente, pero new y delete no coinciden
std::unique_ptr<uint8_t> ptr(new uint8_t[10]);
std::unique_ptr<uint8_t[]> ptr(new uint8_t);
// No es seguro ante excepciones: el compilador puede calcular los parámetros en el siguiente orden:
// 1. Asignar memoria para Foo,
// 2. Construir Foo,
// 3. Llamar a Bar,
// 4. Construir unique_ptr<Foo>.
// Si Bar lanza una excepción, Foo no se destruye, causando una fuga de memoria.
F(unique_ptr<Foo>(new Foo()), Bar());

// Seguro ante excepciones: las llamadas a funciones no se interrumpen.
F(make_unique<Foo>(), Bar());

Excepción std::make_unique no admite deleter personalizado. En escenarios que requieren deleter personalizado, se recomienda implementar una versión personalizada de make_unique en su propio espacio de nombres. Usar new para crear unique_ptr con deleter personalizado es la última opción.

Regla 10.2.4 Use std::make_shared en lugar de new para crear shared_ptr

Razón Usar std::make_shared para crear std::shared_ptr además de las razones de consistencia similares a std::make_unique, también hay factores de rendimiento. std::shared_ptr gestiona dos entidades:

  • Bloque de control (almacena recuento de referencias, deleter, etc.)
  • Objeto gestionado

std::make_shared crea std::shared_ptr, asignará una vez en el montón suficiente memoria para contener el bloque de control y el objeto gestionado. Mientras que usar std::shared_ptr<MyClass>(new MyClass) para crear std::shared_ptr, además de que new MyClass activa una asignación en el montón, el constructor de std::shard_ptr también activa una segunda asignación en el montón, generando gastos adicionales.

Excepción Similar a std::make_unique, std::make_shared no admite deleter personalizado

Lambda

Recomendación 10.3.1 Cuando las funciones no pueden trabajar, elija usar lambda(capturar variables locales, o escribir funciones locales)

Razón Las funciones no pueden capturar variables locales o declararse en un ámbito local; si necesita estas cosas, elija lambda en lugar de functor escrito a mano. Por otro lado, lambda y functor no sobrecargan; si necesita sobrecarga, use funciones. Si tanto lambda como funciones pueden usarse, entonces use funciones; use la herramienta más simple posible.

Ejemplo

// Escribir una función que solo acepte int o string
// -- la sobrecarga es la elección natural
void F(int);
void F(const string&);

// Necesita capturar estado local, o aparece en el ámbito de sentencias o expresiones
// -- lambda es la elección natural
vector<Work> v = LotsOfWork();
for (int taskNum = 0; taskNum < max; ++taskNum) {
    pool.Run([=, &v] {...});
}
pool.Join();

Regla 10.3.1 Evitar usar captura por referencia en lambdas fuera del ámbito local

Razón El uso de lambdas fuera del ámbito local incluye valores de retorno, almacenamiento en el montón, o paso a otros hilos. Los punteros y referencias locales no deberían existir más allá de su ámbito. La captura por referencia en lambdas almacena referencias a objetos locales. Si esto hace que las referencias existan más allá del ciclo de vida de los objetos locales, entonces no se debe usar la captura por referencia.

Ejemplo

// No es bueno
void Foo()
{
    int local = 42;
    // Captura local por referencia.
    // Después de que la función retorne, local ya no existe,
    // por lo tanto, el comportamiento de Process() es indefinido!
    threadPool.QueueWork([&]{ Process(local); });
}

// Bueno
void Foo()
{
    int local = 42;
    // Captura local por valor.
    // Debido a la copia, Process() siempre es válido durante la llamada
    threadPool.QueueWork([=]{ Process(local); });
}

Recomendación 10.3.2 Si captura this, capture explícitamente todas las variables

Razón En una función miembro, [=] parece capturar por valor. Pero debido a que captura implícitamente el puntero this por valor, y puede operar con todos los miembros de datos, los miembros de datos realmente están capturados por referencia, generalmente se recomienda evitarlo. Si realmente necesita hacer esto, especifique explícitamente la captura de this.

Ejemplo

class MyClass {
public:
    void Foo()
    {
        int i = 0;

        auto Lambda = [=]() { Use(i, data_); };   // No es bueno: parece capturar por copia/valor, los miembros de datos están realmente capturados por referencia

        data_ = 42;
        Lambda(); // llama a use(42);
        data_ = 43;
        Lambda(); // llama a use(43);

        auto Lambda2 = [i, this]() { Use(i, data_); }; // Bueno, especifica explícitamente la captura de this, más clara, menos confusión
    }

private:
    int data_ = 0;
};

Recomendación 10.3.3 Evitar usar modos de captura predeterminados

Razón Las expresiones lambda proporcionan dos modos de captura predeterminados: por referencia (&) y por valor (=). La captura por referencia predeterminada captura implícitamente todas las variables locales por referencia, lo que fácilmente lleva a referencias colgantes. En comparación, escribir explícitamente las variables que necesita capturar facilita la verificación del ciclo de vida de los objetos y reduce la posibilidad de errores. La captura por valor predeterminada captura implícitamente el puntero this, y es difícil ver qué variables la función lambda depende. Si hay variables estáticas, también hará que el lector crea erróneamente que lambda copia una variable estática. Por lo tanto, generalmente se debe escribir explícitamente las variables que lambda necesita capturar, en lugar de usar modos de captura predeterminados.

Ejemplo incorrecto

auto func()
{
    int addend = 5;
    static int baseValue = 3;

    return [=]() {  // Realmente solo copia addend
        ++baseValue;    // Modificar afectará el valor de la variable estática
        return baseValue + addend;
    };
}

Ejemplo correcto

auto func()
{
    int addend = 5;
    static int baseValue = 3;

    return [addend, baseValue = baseValue]() mutable {  // Usa la inicialización de captura de C++14 para copiar una variable
        ++baseValue;    // Modificar su propia copia, no afectará el valor de la variable estática
        return baseValue + addend;
    };
}

Referencia: “Effective Modern C++”: Item 31: Avoid default capture modes.

Interfaz

Recomendación 10.4.1 En escenarios que no involucran propiedad, usar T* o T& como parámetros, no punteros inteligentes

Razón

  1. Solo cuando se necesita una transferencia o compartir explícita de propiedad, use punteros inteligentes para transferir o compartir propiedad.
  2. Pasar por puntero inteligente limita al llamador de función a usar punteros inteligentes (como cuando el llamador quiere pasar this).
  3. Pasar punteros inteligentes compartidos de propiedad tiene costos de tiempo de ejecución.

Ejemplo

// Acepta cualquier int*
void F(int*);

// Solo acepta int que desee transferir propiedad
void G(unique_ptr<int>);

// Solo acepta int que desee compartir propiedad
void G(shared_ptr<int>);

// No cambia propiedad, pero necesita un llamador con propiedad específica
void H(const unique_ptr<int>&);

// Acepta cualquier int
void H(int&);

// No es bueno
void F(shared_ptr<Widget>& w)
{
    // ...
    Use(*w); // Solo usa w -- no involucra gestión de ciclo de vida en absoluto
    // ...
};
